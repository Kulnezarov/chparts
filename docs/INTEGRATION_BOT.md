# CHPARTS + Sklad API + Telegram-бот (вариант A)

Витрина **chparts** — это Next.js, который в браузере проксирует запросы на бэкенд через `/api/proxy/...` (см. `next.config.ts`).  
**Бот** (отдельный процесс) должен обращаться **напрямую** к тому же бэкенду, без Next.js:

```
База: {NEXT_PUBLIC_API_BASE_URL}/api/v1
```

Пример: `https://api.example.com/api/v1/public/products`  
(у вас в `.env.local` это `http://194.32.142.253` — без слеша в конце)

## Endpoints, которые использует витрина (публичные)

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/public/products` | Каталог (query: q, category_id, brand_id, limit, offset, sort, …) |
| GET | `/public/products/{id}` | Карточка товара |
| POST | `/public/orders` | Оформление заказа с сайта; ответ: `{ ok, reserve_id }` |
| GET | `/public/categories` | Категории |
| GET | `/public/brands` | Марки |
| GET | `/public/reserves/{reserve_id}` | **Статусы позиций резерва** (нужен для «Мои заказы») — детали в `BACKEND_FASTAPI_PUBLIC_API.md` |
| GET | `/public/orders/{id}` | Запасной вариант, если id совпадает с моделью «order» на бэкенде |

Реализация на фронте: `lib/publicApi.ts`.

## Что сделать на бэкенде (склад), не в CHPARTS

1. Реализовать **GET `/api/v1/public/reserves/{id}`** с позициями и статусами — иначе витрина не подтянет «выдано/отменено».
2. При необходимости добавить **отдельные** маршруты для бота (например `GET /api/v1/bot/...` с `Authorization: Bearer <BOT_API_TOKEN>`), если публичного API мало: список заказов за день, агрегаты продаж — **это делается в FastAPI склада**, не в Next.

## Бот: вариант A (тот же API)

- Бот **не ходит** на `localhost:3000` и не использует `/api/proxy` — это только для браузера.
- Бот ходит на **origin из `NEXT_PUBLIC_API_BASE_URL`**: `fetch('https://.../api/v1/public/...')` из Python/Node **с сервера** (VPS, cron), где нет CORS-проблем.
- **Токен бота** хранить только в окружении бота, не публиковать.

Telegram-бот (менеджер/склад):
- Слать уведомление о новом заказе с сайта (чат/топик) в дополнение к уже существующим дневным отчётам.

Отмена в кабинете/админ API:
- Менеджер обязан выбрать причину из enum; при other — обязателен текстовый комментарий.

Бэкенд:
- После успешного commit в БД вызывать единую отправку уведомлений по событию; идемпотентность, чтобы при ретрае не дублировать одно и то же.
- При необходимости: привязка Telegram к телефону; GET резерва/списка для бота с BOT_API_TOKEN.

Безопасность: TELEGRAM_BOT_TOKEN и ключи в .env, не логировать токены и полные ПД. Тесты: мок отправки, 422 при other без комментария, маппинг причин.
